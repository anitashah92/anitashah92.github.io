Fusion Design System

Building a design system from scratch — on top of MUI — to power a 3-year, 6-application commerce intelligence platform used by Fortune 500 brands.

Role
Senior UX/UI Designer
Timeline
April 2021 – 2024
Tools
Figma, MUI, Nivo Rocks
Scale
50+ components, 6 apps
The problem

Engineers had MUI. Designers had nothing.

When I joined GroupM's Fusion team, the engineering side was already using Material UI (MUI) as its component framework. But there was no design-side system to match it. No shared color tokens, no documented type scale, no Figma component library — just a collection of individual screens built inconsistently across the platform.

The team was also using Sketch, which was creating its own friction. Files weren't easily shareable, components couldn't be linked across documents, and prototyping required separate tooling. Every new screen started from scratch rather than from a reusable foundation.

As the sole designer responsible for every screen across the platform — and with six applications planned — the gap between what engineering had and what design had wasn't sustainable. I needed to build the design side of the system from scratch, get the team onto better tooling, and maintain both as the platform grew over three years.

First: fixing the tooling

Six months to convince one person — and why it was worth it

Before I could build a proper design system, I needed to move the team off Sketch. The problem wasn't Sketch itself — it was that the head of product and platform was deeply comfortable in it and skeptical of switching. Changing design tools in the middle of an active product build felt like a risk he didn't want to take.

I spent six months building the case. I created a formal proposal comparing Figma and Sketch across the dimensions that mattered most to our team: component linking across files, real-time collaboration with engineers, prototyping built in rather than requiring InVision, version history, and browser-based access that didn't require a license per machine.

Sketch — before

Components lived in isolated files with no cross-file linking. Any update to a shared component had to be manually propagated across every file it appeared in.

Real-time collaboration didn't exist — engineers couldn't inspect designs without a separate Zeplin handoff workflow.

Prototyping required InVision, adding a separate tool and export step to every design review.

Mac-only, license-per-seat model created access friction for PM and engineering stakeholders.

Figma — after

Component libraries published from a single source file. Any update to a component propagates everywhere it's used across the entire platform instantly.

Engineers could inspect designs, copy CSS, and leave comments directly in the file — no separate handoff tool needed.

Prototyping built into the same canvas. Design reviews happened in Figma, with stakeholders clicking through flows live.

Browser-based and cross-platform — anyone on the team could view and comment without a paid seat.

The proposal worked. Once the head of product could see the concrete workflow differences laid out side by side, the case was clear. We migrated to Figma and within weeks the benefits were already visible — engineers were inspecting designs directly, components were linked across files, and design reviews became collaborative rather than sequential.

Moving to Figma wasn't just a tool preference. It was the prerequisite for building a real design system — one where a component change propagates everywhere instead of requiring manual updates across dozens of files.
Building the foundations

Tokens first — everything else follows

With Figma established as the team's design tool, I started the system with the foundational layer: color tokens, typography, and spacing. These had to map directly to MUI's token structure so that what engineers implemented matched what design specified — without a translation step in the middle.

Colors

Semantic tokens over raw hex values

Rather than documenting individual hex values, I built a semantic token layer — Primary, Secondary, Neutral, Error, Warning, Success — each with a full shade scale. This meant engineers referenced token names in code, and changing a brand color later required updating one value, not hunting through the codebase.

Typography

A type scale mapped to MUI's system

Fusion used MUI's default typography system as its base, which I documented in Figma with the exact MUI naming conventions (h1–h6, body1, body2, caption, overline). This meant designers and engineers were speaking the same language when specifying text styles — no conversion or guesswork required.

Spacing

8px grid, documented as a constraint

All spacing values — padding, margins, gaps — were built on an 8px base grid, matching MUI's default spacing system. I documented this as a rule in the system rather than leaving it implicit, which meant new components built by design or engineering stayed consistent without explicit sign-off on every value.

Component library

50+ components — built to match MUI, documented to scale

With the token layer in place, I built out the component library in Figma — matching every MUI component the engineering team was using and adding the platform-specific components Fusion needed that MUI didn't cover out of the box.

Each component was documented with all its variants, states, and usage guidelines. The goal wasn't just visual fidelity to MUI — it was making sure a designer picking up a new screen could find the right component, understand when to use it, and hand it off to engineering without ambiguity.

Principle 01

Every state, documented

Default, hover, focus, disabled, error, loading — every interactive component was documented in all states. This eliminated the most common cause of design-engineering gaps: assumptions about what a button looks like when it's disabled, or how an input behaves in an error state.

Principle 02

Variants as properties, not separate frames

Using Figma's component properties system, each component's variants (size, color, state) were exposed as dropdown controls in the properties panel. Designers could configure a button to Primary/Large/Hover without duplicating frames — and without accidentally using an outdated version.

Principle 03

Platform-specific additions alongside MUI

Fusion had components MUI didn't: data cards, score tiles, evaluation tables, segment selectors. These were built from MUI primitives (typography, color tokens, spacing) rather than from scratch — so they inherited the system's consistency without requiring separate documentation of their foundations.

Extending the system — data visualization

Adding charts to a system that wasn't built for them

As Fusion grew, the platform increasingly needed data visualization — treemaps, radar charts, bubble heatmaps, scatter plots, bar charts. MUI didn't include chart components, so the engineering team used Nivo Rocks as the charting library. That meant extending the design system to cover chart patterns that lived outside its original scope.

Rather than treating charts as one-off designs, I documented them as system components: each chart type had a defined color palette drawn from the token layer, a consistent approach to axes and labels, interaction states (hover, tooltip, selected), and usage guidelines for when to use each chart type versus the alternatives.

Bringing charts into the system rather than leaving them as bespoke designs paid off every time a new chart was needed. The color palette, label sizing, tooltip behavior, and empty state were already decided — designers and engineers could focus on the data question, not the visual language.

Theming & workspaces

One system, multiple identities

Fusion was used across GroupM's different operating companies — Mindshare, Wavemaker, and others — each with their own brand identity. A single visual theme wouldn't work. The system needed to support multiple themes without duplicating components.

I built a theming layer on top of the token system. Each Op-Co had its own theme file that overrode the semantic color tokens — primary brand color, accent, surface colors — while inheriting everything else from the base system. The components themselves never changed; only the token values they referenced did.

This meant onboarding a new Op-Co didn't require rebuilding any screens. A new theme file plus their brand tokens, and the entire platform instantly reflected their identity. What could have been weeks of work per client became a few hours of configuration.

Maintaining the system over 3 years

A design system is never done

Building the system was the first challenge. Keeping it accurate, useful, and adopted over three years while the platform was actively growing was the harder one.

Year 1 — 2021
Foundations + migration
Sketch → Figma migration, token layer, core component library built to match MUI. System adopted for all new screens from day one.
Year 2 — 2022
Data viz additions
Charts brought into the system as Insights was built. Nivo Rocks components documented alongside MUI components. Chart usage guidelines written to prevent misuse.
Year 3 — 2023
Theming + scale
Theming layer added for Op-Co workspaces. User panel for Fusion & Copilot maintained at 200+ participants. System extended to cover 6 applications across the platform.

The most important maintenance decision was treating the system as a living document rather than a deliverable. Every new component built for a specific feature was reviewed for whether it belonged in the shared library. Every inconsistency caught in a design review became a system update rather than a one-off fix. That discipline was what kept the system relevant rather than stale.

Color values that didn't need to be re-specified in each new screen. Themes that didn't require new component designs. The compounding value of a design system only becomes clear over time — and three years of building on top of it made that visible.

Reflections

What this project taught me

What I learned

Building a design system as a solo designer taught me that the hardest part isn't the component architecture — it's the communication and adoption work. A system that lives in a Figma file nobody opens isn't a system; it's a library. Making it useful meant making it discoverable, making the naming intuitive, and being the person who consistently pointed back to it when shortcuts emerged in design reviews.

The Sketch → Figma migration also taught me how to move an organization through a tool change. The proposal wasn't just a feature comparison — it was a business case. Framing the switch in terms of engineering handoff efficiency and reduced tooling costs landed better than any design argument I could have made.

What I'd do with more time

With more resources, I would have invested in a documentation site — a living Storybook or Zeroheight instance where component usage, code snippets, and design guidelines lived alongside each other. The Figma file was the source of truth for design, but engineers had to cross-reference it manually. A shared doc site would have closed that loop.

I also would have run a formal design system audit annually — measuring component adoption across the platform, flagging components that had drifted from spec, and retiring ones no longer in use. The system stayed accurate through vigilance rather than process, and a more structured audit cycle would have made that more scalable as the team grew.
